# Data Engines 工作计划

## 1. 目标对齐

基于 `data_engines/task.md`，本阶段目标不是继续零散增加数据抓取脚本，而是建设一套面向量化 Agent 的统一金融数据资产层：

- 统一描述多源异构数据（量价 / 财报 / 公告 PDF / 宏观 / 情绪）
- 统一对外接口与返回契约，优先采用结构化 `JSON` 语义
- 屏蔽底层数据源差异、缓存、限频、切源与溯源细节
- 为后续接入更多免费/商业数据源预留标准化扩展入口

## 2. 当前已确认决策

- 工作目录统一使用 `data_engines/`
- 统一返回以 `JSON-first` 为主，`DataFrame` 作为内部实现或可选输出层
- 商业数据源（如 `Wind`、`iFinD`）当前阶段只做“方案级纳入与接口预留”，不做真实适配
- `data_engines/` 不以现有 `data_providers/` 为实现基线，按新方案独立设计与落地；旧实现最多作为调研参考，不作为结构约束

## 3. 现状基线

当前阶段的“基线”不再定义为仓库里已有的 `data_providers/` 实现，而是定义为 `data_engines/task.md` 所要求的目标能力边界。也就是说：

- 不沿用旧的数据访问层结构作为默认方案
- 不让旧的 `DataFrame-first` 契约限制新的统一资产模型
- 以“面向 Agent 的清晰数据引擎架构”重新定义目录、模型、接口、运行时能力与 Provider 接入规范
- 旧目录中的实现仅可用于参考数据源特性、字段口径与踩坑经验，不作为新引擎的架构前提

当前主要缺口：

- 缺少从 `JSON-first` 出发的统一资产模型与目录结构
- 缺少覆盖量价 / 财报 / 公告 / 宏观 / 情绪的统一对象语义
- 缺少独立于具体 Provider 的接口层、编排层与运行时治理层设计
- 缺少完整的数据源选型矩阵、接入优先级与标准化测试体系

当前已沉淀的草案基础：

- `data_engines/task.md` 已补齐 9 个核心对象的中文释义与字段级定义草案
- `data_engines/task.md` 已补齐通用响应对象 `records/meta/trace/quality/errors` 的职责说明
- `data_engines/task.md` 已补齐 `BaseProvider`、`BaseNormalizer`、`BaseDocumentParser`、`BaseResolver` 的最小职责接口草案
- `data_engines/task.md` 已补齐行情、文档、财报 / 宏观 / 情绪三类标准响应 `JSON` 样例

## 4. 分阶段执行计划

| 阶段 | 主题 | 目标 | 核心产出 | 状态 |
|---|---|---|---|---|
| Phase 1 | 数据源调研与选型 | 梳理免费/商业/情绪/宏观数据源差异，形成接入优先级 | 数据源调研报告、能力对比矩阵、接入建议、正式调研结论 | 已完成 |
| Phase 2 | 统一资产模型设计 | 定义统一元数据、实体结构、引用结构、质量结构 | 统一 Schema 草案、字段映射表、数据集分层、字段级对象草案 | 进行中 |
| Phase 3 | 统一接口与抽象基类 | 定义 Request/Response、Provider/Normalizer/Parser/Base 接口 | 接口规范、扩展规范、错误码与元数据约定、标准响应样例 | 进行中 |
| Phase 4 | 首批 Provider 适配策略 | 确定首批优先接入源与新引擎落地范围 | Provider 接入计划、适配顺序、实现边界 | 未开始 |
| Phase 5 | 测试与验收体系 | 建立 schema/provider/normalizer 层面的测试规范 | 测试清单、样例、验收标准 | 未开始 |
| Phase 6 | 稳定性与可靠性治理 | 完善可信度、缓存、限频、回退、监控设计 | 稳定性方案、降级策略、指标定义 | 未开始 |

## 5. 每阶段拆解

### Phase 1｜数据源调研与选型

#### 目标

建立统一的数据源认知基础，为后续 Schema 与 Provider 抽象提供约束。

#### 拆解任务

1. 梳理数据域分类：
   - 量价
   - 财报
   - 公告 / PDF / 文档
   - 宏观
   - 情绪 / 热度 / 舆情
2. 盘点候选数据源：
   - 免费 / 开源：`AkShare`、`Tushare`、`BaoStock`
   - 商业：`Wind`、`iFinD`（方案纳入，不做适配）
   - 文档 / 公告：`Cninfo`、东方财富、交易所官方接口
   - 情绪 / 热度：`微信指数`、`雪球`
   - 宏观：国家统计局、人民银行、财经日历
   - 研究 / 平台型：`聚宽`
   - 交易所官方：上交所、深交所、港交所、纽交所
3. 建立对比维度：
   - 覆盖市场
   - 覆盖数据域
   - 更新频率 / 实时性
   - 历史深度
   - 授权与成本
   - 调用方式
   - 限频策略
   - 字段规范程度
   - 稳定性 / 可用性
   - 适合场景（研究 / 回测 / 生产）
4. 形成接入优先级：
   - 优先真实接入源
   - 仅预留接口源
   - 用于交叉验证的对照源

#### 验收标准

- 每类数据域至少有 2 个候选源被比较
- 免费源与商业源的边界清晰
- 输出可以直接支撑后续统一 Schema 设计
- 已形成主源 / 备源 / 对照源 / 预留源的正式建议
- 已形成对统一 `JSON-first` Schema 的正式约束清单

### Phase 2｜统一资产模型设计

#### 目标

把“取数接口”升级为“统一信息资产表达”。

#### 拆解任务

1. 定义核心实体：
   - `Instrument`
   - `Dataset`
   - `Record`
   - `Document`
   - `Citation`
   - `Event`
   - `Sentiment`
   - `QualityProfile`
   - `SourceTrace`
2. 设计统一元数据：
   - 数据集标识
   - 数据源标识
   - 抓取时间
   - 生效时间 / 发布时间 / 观测时间
   - 市场 / 交易所 / 币种
   - 缓存信息
   - 可信度 / 完整度 / 新鲜度
3. 定义各类数据的标准结构：
   - 量价数据结构
   - 财报数据结构（raw / normalized / statement-level）
   - 公告 PDF 结构（文档级 / 页级 / 引用级）
   - 宏观序列结构
   - 情绪事件 / 指数结构
4. 定义多源映射关系：
   - 同一资产的源间字段映射
   - 同一时间点的多源对齐规则
   - 冲突字段的优先级与保留策略
5. 产出字段级定义草案：
   - 为 9 个核心对象给出最小字段集合
   - 明确必填 / 可选 / 可空字段边界
   - 明确对象间引用关系

#### 验收标准

- Schema 能覆盖 `task.md` 里要求的全部数据域
- 支持 `JSON` 输出语义，并把 `DataFrame` 降为内部适配层或可选导出层
- 每类数据域都有字段级映射说明
- 核心对象具备可落地的字段级定义草案，可直接进入 `JSON Schema` 设计

### Phase 3｜统一接口与抽象基类

#### 目标

建立“任何新数据源都能按同一标准接入”的工程规范。

#### 拆解任务

1. 定义统一请求对象：
   - 通用 `BaseRequest`
   - 各领域子请求对象
2. 定义统一响应对象：
   - `records`
   - `meta`
   - `trace`
   - `quality`
   - `errors`
3. 定义抽象接口：
   - `BaseProvider`
   - `BaseNormalizer`
   - `BaseDocumentParser`
   - `BaseResolver`
4. 定义运行时能力：
   - 缓存
   - 限频
   - 重试
   - 切源
   - 溯源
5. 定义接入流程模板：
   - 能力声明
   - 字段映射
   - 返回契约检查
   - 样例与测试要求
6. 产出最小接口草案：
   - 明确 `BaseProvider` / `BaseNormalizer` / `BaseDocumentParser` / `BaseResolver` 的方法边界
   - 明确响应外壳与多数据域 `records` 载荷复用方式
   - 提供行情 / 文档 / 财报 / 宏观 / 情绪的标准响应样例

#### 验收标准

- 新 Provider 接入步骤明确且可复制
- 响应结构适合 Agent 直接解析
- 错误类型与降级语义可标准化
- 基类具备最小职责接口草案，可直接进入代码骨架实现
- 响应外壳具备跨数据域复用样例，可直接约束后续 Provider 输出

### Phase 4｜首批 Provider 适配策略

#### 目标

在统一标准下确定首批真实接入实现顺序。

#### 建议顺序

1. `AkShare`：量价 / 基础行情
2. `东方财富`：财报 / 基本面聚合
3. `巨潮 / CNInfo`：公告 PDF / 文档元数据
4. `Tushare` / `BaoStock`：作为对照源与补充源
5. 交易所官方接口：作为官方口径与公告披露对照源
6. 商业源：仅定义能力接口和占位 Provider

#### 验收标准

- 每个优先源都有明确定位：主源 / 备源 / 对照源 / 预留源
- 新引擎首批实现边界明确，且不被旧结构绑定

### Phase 5｜测试与验收体系

#### 目标

保证统一模型不是“纸面设计”，而是可验证、可迭代的。

#### 拆解任务

1. Schema 层测试
2. Symbol / Resolver 测试
3. Provider contract test
4. Normalizer 纯函数测试
5. Document parser 测试
6. 缓存 / 限频 / 重试测试
7. 端到端样例用例

#### 验收标准

- 每类核心能力至少有一组正向样例与异常样例
- Provider 输出满足统一契约检查

### Phase 6｜稳定性与可靠性治理

#### 目标

确保数据资产“可信、稳定、可追溯、不过时”。

#### 拆解任务

1. 多源回退与优先级
2. 数据质量评分与可信度标注
3. 分级缓存策略
4. 限频隔离与重试策略
5. 空数据 / 脏数据 / 冲突数据处理
6. 监控指标与日志标准

#### 验收标准

- 关键失败场景都有预案
- 任意返回结果都可以追溯来源与质量状态

## 6. 当前优先执行顺序

接下来按以下顺序推进：

1. 将已完成的 `Phase 1` 正式调研结论回写到 `Phase 2` 统一资产模型文档
2. 把已写入 `task.md` 的对象字段草案整理为独立的 `Phase 2` 统一资产模型草案
3. 把已写入 `task.md` 的响应样例与基类接口草案整理为独立的 `Phase 3` 接口规范草案
4. 最后在 `data_engines/` 下按新架构独立落地首批引擎骨架与 Provider 适配
