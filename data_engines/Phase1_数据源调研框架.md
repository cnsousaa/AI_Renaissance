# Phase 1 数据源正式调研

## 1. 调研结论摘要

本轮调研基于 2026-05 可公开访问的官网、文档页、入口页与公开说明完成，目标不是给出“唯一数据源”，而是形成 `data_engines/` 第一阶段的正式接入判断。

核心结论如下：

1. **免费主干组合已经明确**：
   - 量价 / 基础行情主干优先考虑 `AkShare + Tushare`
   - A 股基础对照源优先保留 `BaoStock`
   - 公告 / PDF 主源优先选择 `巨潮 / CNInfo`
   - 财报 / 行情聚合主源优先选择 `东方财富`
   - 宏观官方主源优先选择 `国家统计局 + 人民银行`
2. **平台型与商业型数据源定位清晰**：
   - `聚宽` 更适合作为研究平台型补充与接口设计参考，不宜直接充当底层统一主源
   - `Wind`、`iFinD` 覆盖广、能力强，但当前阶段只做“能力纳入 + 接口预留”
3. **交易所官方接口是权威对照源，不是首选聚合主源**：
   - 上交所 / 深交所 / 港交所 / 纽交所更适合作为公告披露、上市目录、官方规则、技术接口和口径校验来源
   - 不适合作为第一阶段统一数据引擎的唯一主抓取层
4. **情绪与热度信号的工程风险高于传统行情与财报**：
   - `微信指数` 具备研究价值，但公开可编程能力弱、接入稳定性风险高
   - `雪球` 与 `东方财富` 具备社区热度价值，但反爬、页面变动、授权边界都更敏感
5. **统一模型必须坚持 `JSON-first + raw/normalized 双层`**：
   - 免费聚合源与官方源、文档源、事件源之间口径差异很大
   - 第一阶段必须强制保留 `raw_values/source_trace/quality/errors`，不能只保留平铺后的标准字段

---

## 2. 调研方法与证据边界

### 2.1 调研方法

- 以官网、官方文档、公开帮助页、公开导航页为优先证据
- 对每个来源从数据域覆盖、市场覆盖、访问方式、稳定性、授权与接入成本进行评估
- 不以二手教程或非官方博客作为主要依据

### 2.2 证据边界

- 本报告属于 **Phase 1 正式调研**，可以支持方案与接口设计
- 但仍不等于生产可用性验收，尤其是以下能力需要 Phase 4 再验证：
  - 字段级覆盖完整性
  - 历史深度真实性
  - 限频与稳定性上限
  - 批量抓取可持续性
  - 授权与商用合规边界

### 2.3 特别说明

- `BaoStock`、`iFinD`、`微信指数` 等来源的公开网页存在 JS 渲染、页面不可直接抽取或入口较弱的问题，因此本轮对这几类来源给出的是**正式方案级判断**，不是接口级最终验收结论

## 4. 按数据域拆分的正式判断

### 4.1 量价与行情

#### 正式结论

- 第一阶段推荐 **`AkShare + Tushare` 双主干**
- `BaoStock` 作为 **A 股基础行情对照源**
- `东方财富` 作为 **聚合补充源**
- 各交易所官方页面作为 **披露与口径对照源**，不直接作为统一行情主源

#### 对比判断

| 来源 | 结论 |
|---|---|
| `AkShare` | 覆盖广，适合快速支持 A/H/美股与多资产原型；但需把多来源异构、限频与字段脏口径留给引擎治理层处理 |
| `Tushare` | 结构化更强，适合回测与研究；应重点评估权限等级、接口额度与字段可得性 |
| `BaoStock` | 能力相对轻，但对 A 股基础行情验证有价值；适合做低成本核对层 |
| `聚宽` | 适合作为研究平台能力参考，不建议在 Phase 1 直接作为统一抓取底层 |
| `东方财富` | 适合作为行情补充与部分指标补足，但需防止页面接口漂移 |
| 交易所官方 | 权威，但更适合做公告、规则与上市信息校验，不适合直接做统一行情主抓层 |

#### 接入建议

- `主源`：`AkShare`、`Tushare`
- `备源`：`东方财富`
- `对照源`：`BaoStock`
- `官方对照源`：上交所、深交所、港交所、纽交所

### 4.2 财报与基本面

#### 正式结论

- 第一阶段推荐 **`东方财富` 作为财报聚合主源**
- `Tushare` 作为 **结构化补充源**
- `Wind`、`iFinD` 保留为 **商业能力上限参考**
- 交易所官方披露页保留为 **原始公告与正式口径校验源**

#### 对比判断

| 来源 | 结论 |
|---|---|
| `东方财富` | 财报、资金流、研报、股东股本等聚合能力强，中文场景成熟，适合作为第一阶段基本面主源 |
| `Tushare` | 结构化程度好，适合与东方财富形成交叉映射 |
| `聚宽` | 更适合作为研究型接口参考，而不是统一引擎底层主财报源 |
| `Wind` / `iFinD` | 覆盖广且专业，适合定义未来商业接口能力模型，但当前阶段不做真实适配 |
| 交易所官方 | 适合做原始披露与报告期核验，不适合独立承担 normalized 财报主源职责 |

#### 接入建议

- `主源`：`东方财富`
- `备源`：`Tushare`
- `对照源`：交易所官方披露页
- `预留源`：`Wind`、`iFinD`

### 4.3 公告与文档资产

#### 正式结论

- 第一阶段推荐 **`巨潮 / CNInfo` 作为公告与 PDF 主源**
- `东方财富` 作为 **公告检索与聚合补充源**
- 各交易所官方站点作为 **披露校验源**

#### 对比判断

| 来源 | 结论 |
|---|---|
| `巨潮 / CNInfo` | 覆盖公告、年报、问询函、纪律处分、个股资料与数据服务入口，最适合承担 A 股公告文档主源 |
| `东方财富` | 可补公告专题与日历型入口，但原文与溯源权威性不应强于 CNInfo |
| 上交所 / 深交所 | 适合做披露校验与公告分市场补充 |
| 港交所 / 纽交所 | 适合做港美股上市与披露对照，不宜与 A 股公告源混为一层接口 |

#### 接入建议

- `主源`：`巨潮 / CNInfo`
- `备源`：`东方财富`
- `对照源`：上交所 / 深交所 / 港交所 / 纽交所

### 4.4 宏观与财经事件

#### 正式结论

- 第一阶段推荐 **`国家统计局 + 人民银行` 作为宏观官方主源组合**
- `财经日历` 作为 **事件流补充源**
- `东方财富` 可作为 **宏观专题聚合补充源**
- `Wind`、`iFinD` 仅做未来商业宏观接口预留

#### 对比判断

| 来源 | 结论 |
|---|---|
| `国家统计局` | 最适合作为官方统计时间序列主源，但程序化接入需额外处理 JS 页面或后台接口 |
| `人民银行` | 最适合作为货币金融、LPR、汇率、政策公告主源 |
| `财经日历` | 更适合做事件对象 `Event`，而非单纯时间序列主源 |
| `东方财富` | 宏观专题丰富，但不应替代官方口径 |
| `Wind` / `iFinD` | 商业能力强，但本阶段只定义预留能力 |

#### 接入建议

- `主源`：`国家统计局`、`人民银行`
- `备源`：`东方财富` 宏观专题
- `补充事件源`：`财经日历`
- `预留源`：`Wind`、`iFinD`

### 4.5 情绪、热度与舆情代理信号

#### 正式结论

- 第一阶段情绪层以 **“可接入性风险”优先约束设计**，不应过早承诺稳定生产能力
- `微信指数` 可作为 **热度代理信号候选**，但应标记为高接入风险
- `雪球` 与 `东方财富` 更适合作为 **社区 / 讨论度补充源**

#### 对比判断

| 来源 | 结论 |
|---|---|
| `微信指数` | 中文热度代理价值高，但公开程序化能力弱，必须在模型中显式标记 `quality` 与 `source_trace` 风险 |
| `雪球` | 适合主题热度、社区讨论、舆情代理；结构化弱、反爬风险高 |
| `东方财富` | 可作为股吧 / 财经专题补充，但同样面临页面与授权边界问题 |

#### 接入建议

- `候选主情绪源`：`微信指数`（仅方案级）
- `补充源`：`雪球`、`东方财富`
- `策略`：情绪层先做抽象结构，真实接入延后于行情 / 财报 / 文档 / 宏观

## 6. 对统一 Schema 设计的正式约束

### 6.1 统一响应必须支持多数据形态

通过本轮调研可确认，统一响应不能只服务二维表，至少要支持：

- `timeseries`：量价、宏观、热度指数
- `statement`：财报三表与派生指标
- `document`：公告 PDF、研报、政策原文
- `citation`：页级 / 段级引用定位
- `event`：宏观发布时间、财报披露、政策事件
- `sentiment`：热度值、讨论度、主题情绪

### 6.2 必须保留 `raw + normalized + trace + quality`

原因如下：

- 免费聚合源与官方源字段口径差异大
- 文档与情绪数据天然不是标准表格
- 宏观与事件存在“发布时间 / 生效时间 / 观测时间”多时点并存
- 情绪与社区类数据必须保留质量与可解释性标记

因此统一模型必须显式保留：

- `raw_values`
- `source_trace`
- `quality`
- `errors`

### 6.3 必须支持 `instrument_id` 可空场景

调研已确认以下数据并不总是绑定证券标的：

- 宏观指标
- 宏观事件
- 主题热度
- 政策文件

因此 `Record`、`Event`、`Sentiment` 需要允许 `instrument_id = null`，同时通过 `topic`、`indicator_code`、`document_type`、`market` 等字段保持语义可识别。

### 6.4 必须支持官方源与聚合源并存

第一阶段不应追求所有数据都来自官方，现实可行路径是：

- 聚合源承担“研究效率”
- 官方源承担“权威校验与文档溯源”
- 商业源承担“未来接口上限预留”

因此 `SourceTrace` 必须能表达：

- 直接主源
- 备源切换
- 官方对照
- 商业预留映射
